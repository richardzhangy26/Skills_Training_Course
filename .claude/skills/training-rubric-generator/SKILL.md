# 训练评价标准生成器

根据实训任务文档，生成详细的评价标准，为大模型和人类评判提供清晰的评价框架。

## 使用时机

当用户需要为能力训练创建完整的评价标准时使用此 skill，典型场景包括:
- 用户提供了实训任务文档，需要生成或完善评价标准
- 用户需要为大模型评判提供详细的评价要求和示例
- 用户想要为训练创建规范化的评分体系
- 用户提到"生成评价标准"、"创建评分项"、"完善评价体系"等关键词

## Instructions

### 第一步: 读取并分析实训任务文档

1. 通过 Read 工具读取用户提供的实训任务文档
2. 提取以下关键信息:
   - **任务目标**: 学生需要达成的能力目标
   - **任务描述**: 智能体的人设、任务流程、各阶段要求
   - **现有评价标准**: 如文档中已有评价标准表格，完整提取

### 第二步: 提取或生成评价标准

#### 方案A: 文档中已有评价标准
1. 从表格中提取所有评分项（名称、描述、分值）
2. 保留原始信息，进行完善和扩展

#### 方案B: 文档中没有评价标准
根据任务目标和描述，自动生成评价标准：

**生成策略**:
- 识别任务的核心能力要求
- 将能力需求转化为可评估的评分项
- 建议 3-5 个评分项，总分 10-20 分
- 每个评分项 2-5 分不等

**通用评分项参考**:
- **知识掌握**: 学生是否理解核心知识点
- **实际应用**: 学生是否能应用知识解决问题
- **逻辑思维**: 学生的分析和推理是否清晰
- **表达能力**: 学生的表述是否清晰、完整、专业
- **态度与合作**: 学生的学习态度、配合度等（如适用）

### 第三步: 生成详细的评价要求

对于每个评分项，生成以下内容：

#### 1. 评分项基本信息
```
名称: [评分项名称]
满分值: [0-10的分值]
```

#### 2. 评价描述（面向用户显示）
简明扼要的描述，说明该项目评价的是什么（50-100字）

**示例**:
- 好的描述: "考查学生是否准确理解了离心泵的工作原理，能否识别汽蚀现象的特征，并能基于现象分析根本原因。"
- 差的描述: "知识掌握"（太笼统，缺少具体信息）

#### 3. 大模型评判参考要求
详细列出大模型在评判时应该参考的标准（200-300字），包括：
- 评判的核心要素
- 需要识别的关键内容
- 常见的错误或不足
- 评分逻辑（例如：必须提到X要点才能得满分）

**示例框架**:
```
## 评判要点
1. 核心知识点识别
   - 学生是否提到了[关键概念1]
   - 学生是否准确解释了[关键概念2]

2. 应用能力评估
   - 学生是否能将理论知识与现场现象结合
   - 学生的分析逻辑是否正确

3. 常见不足
   - 混淆概念A和概念B
   - 遗漏关键现象的描述
   - 原因分析不深入

## 评分逻辑
- 满分(5分): 准确理解并完整解释了核心概念，论证严密
- 良好(4分): 基本理解，解释较为准确，少量逻辑不严密
- 中等(3分): 理解不够深入，解释有偏差，逻辑不够清晰
- 不足(2分): 理解有误，解释错误，逻辑混乱
- 缺陷(1分): 严重错误或完全未能理解
```

#### 4. 正例（优秀回答示例）
提供 1-2 个高质量的回答示例，展示满分或高分应该是什么样：
- 示例答案（50-150字）
- 为什么这是好的回答（指出优点）

#### 5. 反例（不足回答示例）
提供 1-2 个不足的回答示例，展示常见错误或低分情况：
- 示例答案（50-150字）
- 这个回答存在的问题（指出缺陷）

**反例选择原则**:
- 不要选择完全错误的答案，而是选择"看起来有道理但有问题"的答案
- 选择真实可能发生的错误，帮助大模型识别边界情况
- 提供至少 2 种常见的不足类型

### 第四步: 生成规范化的JSON配置

整合所有评价标准信息，生成标准的JSON格式配置：

```json
{
  "rubricName": "评价标准名称",
  "totalScore": 总满分,
  "description": "评价标准的整体说明（100字以内）",
  "scoringItems": [
    {
      "id": "item_001",
      "name": "评分项名称",
      "maxScore": 5,
      "userDescription": "面向用户的评价描述（50-100字）",
      "aiGuidelines": {
        "description": "大模型评判参考要求（200-300字）",
        "keyPoints": [
          "关键评判点1",
          "关键评判点2"
        ],
        "commonMistakes": [
          "常见错误1",
          "常见错误2"
        ],
        "scoringLogic": {
          "fullScore": "满分5分的标准说明",
          "goodScore": "良好4分的标准说明",
          "mediumScore": "中等3分的标准说明",
          "lowScore": "不足2分的标准说明",
          "poorScore": "缺陷1分的标准说明"
        }
      },
      "examples": {
        "excellent": {
          "answer": "优秀回答示例",
          "explanation": "为什么这是好回答的说明"
        },
        "poor": [
          {
            "answer": "不足回答示例1",
            "explanation": "存在的问题说明1"
          },
          {
            "answer": "不足回答示例2",
            "explanation": "存在的问题说明2"
          }
        ]
      }
    }
  ],
  "metadata": {
    "createdAt": "创建时间",
    "source": "源文档路径",
    "generationMethod": "extracted或generated"
  }
}
```

### 第五步: 保存配置和生成输出

1. 在实训任务文档所在目录创建以任务名称命名的文件夹（如果不存在）
2. 将生成的评价标准保存为：
   - `评价标准.json` - 完整的规范化配置
   - `评价标准.md` - 人类可读的Markdown格式
3. 向用户输出：
   - 评价标准摘要
   - 文件保存路径
   - 可用于直接的建议

## Important Notes

1. **准确提取信息**: 如文档中已有评价标准，必须完整、准确地提取，保留原意
2. **详细的评判指南**: 大模型评判参考要求必须足够详细，包括关键点、常见错误、评分逻辑
3. **真实的示例**: 正反例必须是真实可能发生的情况，不要编造不合理的例子
4. **完整性**: 每个评分项都要包含正例和至少2个反例
5. **规范化格式**: JSON格式需要保持一致，便于系统解析
6. **分值合理性**: 总分应该在 10-20 分之间，各项分值应该合理分配
7. **评价对称性**: 对于多个评分项，评判指南的详细程度应该相当

## Error Handling

如果遇到以下情况，应该：

1. **文档中评价标准不清晰**: 询问用户是否需要生成或直接提取
2. **无法理解任务要求**: 要求用户提供更详细的任务描述
3. **评分项过多或过少**: 建议合理的数量（3-5个），并说明理由
4. **示例生成困难**: 使用通用的、高质量的示例来说明
